# Speckit Integration for Infrahub Template

**Created**: 2026-03-17
**Status**: Draft

## Summary

Add GitHub Spec Kit to the Infrahub Copier template so that every generated repository ships with spec-driven development tooling pre-configured with Infrahub best practices. The speckit layer acts as a workflow router — directing AI agents to the appropriate Infrahub skill for each task type, regardless of which AI tool the customer uses.

## Context

- This is a Copier template (`copier.yml`) that scaffolds Infrahub repositories
- Opsmill maintains 6 Infrahub skills as a Claude Code plugin: schema-creator, object-creator, check-creator, generator-creator, transform-creator, menu-creator
- Customers may use Claude, Copilot, Cursor, or other AI agents
- Speckit provides an agent-agnostic framework for encoding development workflows

## Approach

**Pre-baked `.specify/` directory** — always included in generated repos (no Copier conditional). Generated by running `specify init --here` and then customizing the scaffolding with Infrahub-specific content.

### Implementation Notes

- The constitution template at `.specify/memory/constitution.md` ships as a generic placeholder from `specify init`. It **must be replaced** with the Infrahub-specific content described below.
- The `.specify/specs/` directory does not exist after `specify init` — it **must be created** with a `.gitkeep`.
- The `.specify/extensions/.cache/` directory is created by `specify init` for the extension catalog. It is preserved as-is.
- Since this is a Copier template, `.specify/` files must not contain unescaped `{{ }}` or `{% %}` syntax, or they must be listed in `copier.yml`'s `_exclude` to prevent Copier from interpreting them as Jinja2. The speckit files do not contain such syntax.
- Check files (`checks/*.py`) are registered in `.infrahub.yml` and placed at the repository root level — there is no Copier conditional for checks since they are part of the base Python project structure.

## Design

### 1. Constitution (`constitution.md`)

The constitution is filled with Infrahub project principles and development standards. It serves as the immutable reference that AI agents always consult.

#### Core Principles

**I. Schema-First Development**
All infrastructure modeling starts with schema design. Conventions:
- PascalCase for node and generic `kind` names (e.g., `DcimDevice`, `IpamIPAddress`)
- snake_case for attribute and relationship names
- Always include `human_friendly_id` on nodes
- Set `identifier` on bidirectional relationships
- Use generics for shared attributes across multiple node types
- Use `InfraGenericDevice` or similar generics for device hierarchies

**II. Validate Before Load**
Always run `infrahubctl schema check` before loading schemas into Infrahub. Never skip validation — schema errors caught early prevent data corruption.

**III. Skill-Driven Workflows**
Use the appropriate Infrahub skill for each task type. The workflow routing table below maps tasks to skills. If the AI agent has Infrahub skills installed, invoke them directly. If not, follow the patterns and conventions described in the constitution and templates.

**IV. Schema Library First**
Before creating custom schemas, check `opsmill/schema-library` for standard models. Use `invoke schema-library-get` to download and integrate. Extend standard models via generics rather than duplicating them.

**V. Code Quality Standards**
- Python 3.11+ required
- Ruff for formatting and linting (line length: 120, max complexity: 17)
- Mypy for type checking (strict mode)
- yamllint for YAML files (max line length: 140)
- All code must pass `invoke lint` before commit

#### Workflow Routing Table

| Task | Skill | Key Files |
|------|-------|-----------|
| Design or modify data models | `infrahub:schema-creator` | `schemas/*.yml` |
| Populate infrastructure data | `infrahub:object-creator` | `objects/*.yml` |
| Build validation checks | `infrahub:check-creator` | `checks/*.py`, `.infrahub.yml` |
| Create design-driven generators | `infrahub:generator-creator` | `generators/*.py`, `queries/*.gql`, `.infrahub.yml` |
| Build data transforms or configs | `infrahub:transform-creator` | `transforms/*.py`, `transforms/templates/*.j2`, `queries/*.gql`, `.infrahub.yml` |
| Customize UI navigation | `infrahub:menu-creator` | `menus/*.yml` |

#### Cross-Cutting Conventions

- **GraphQL queries** — all checks, generators, and transforms require GraphQL queries in `queries/*.gql`. Follow the patterns in the `infrahub:schema-creator` and shared `common/graphql-queries.md` reference.
- **`.infrahub.yml` manifest** — checks, generators, transforms, and artifacts must be registered here. Loading order: schemas → queries → objects → Python → Jinja2 → artifacts.

#### Development Workflow

1. **Check schema library** — run `invoke schema-library-get` and check for existing standard models before creating custom schemas
2. **Schema first** — define or update the data model
3. **Validate** — run `infrahubctl schema check`
4. **Load** — run `invoke load-schema`
5. **Populate** — create object data files and run `invoke load-objects`
6. **Extend** — add checks, generators, transforms, menus as needed
7. **Test** — run `invoke test` for integration tests
8. **Lint** — run `invoke lint` before committing

#### Governance

Constitution principles are enforced in all specs and plans. Deviations must be documented in the Complexity Tracking table of the implementation plan with justification.

### 2. Workflow-Specific Spec Templates

Six templates placed in `.specify/templates/overrides/` for Infrahub-specific workflows. These are **not overrides of `spec-template.md`** in the speckit resolution sense (since they have different names). Instead, they are reference templates that agents or users select manually when starting a new feature. The constitution's workflow routing table directs agents to the appropriate template by name.

Each template:
- Follows the same structure as `spec-template.md` (User Scenarios, Requirements, Success Criteria, Edge Cases) but with domain-specific examples and guidance pre-filled
- Directs the agent to invoke the corresponding skill
- Includes domain-specific user story patterns and acceptance criteria
- References Infrahub-specific file paths and conventions

#### spec-schema-template.md
For schema design tasks. Guides through:
- Node and generic definition
- Attribute and relationship design
- Hierarchy planning
- Validation requirements
- Migration considerations (state: absent for removals)

#### spec-objects-template.md
For data population tasks. Guides through:
- Object file structure (apiVersion, kind, spec)
- Dependency ordering between files
- Relationship references via human_friendly_id
- Range expansion for interfaces ([N-M] syntax)
- Component children patterns

#### spec-check-template.md
For validation check tasks. Guides through:
- Global vs targeted check architecture
- GraphQL query design for validation data
- Python class structure (InfrahubCheck)
- Registration in .infrahub.yml
- Error and info message patterns

#### spec-generator-template.md
For generator tasks. Guides through:
- Design object identification (what triggers generation)
- Target group configuration
- Idempotent create-or-update patterns (allow_upsert)
- Tracking and cleanup (delete_unused_nodes)
- Registration in .infrahub.yml

#### spec-transform-template.md
For transform tasks. Guides through:
- Transform type selection (Python vs Jinja2 vs hybrid)
- GraphQL query design for source data
- Output format specification
- Artifact definition for automated output
- Template organization in transforms/templates/

#### spec-menu-template.md
For navigation customization. Guides through:
- Menu hierarchy design
- Icon selection (MDI icon set)
- Schema node mapping
- Group vs item distinction
- Schema integration (include_in_menu)

### 3. No Agent-Specific Files

The `.github/` copilot-specific agent/prompt files generated by `specify init` are removed. The speckit content is agent-agnostic — the constitution and templates work with any AI tool.

### 4. Generic Templates Preserved

The standard speckit templates (`plan-template.md`, `tasks-template.md`, `checklist-template.md`) are kept as-is from the upstream scaffolding. They are generic enough to work for any Infrahub workflow.

## Out of Scope

- Agent-specific command files (Claude commands, Copilot prompts, etc.)
- Copier conditionals for speckit — it is always included
- Upstream speckit update mechanism — this is a snapshot, not a sync
- The `infrahub-debug` skill — that is an internal tool

## Success Criteria

- Generated repos include a fully populated `.specify/` directory
- Constitution contains Infrahub conventions, skill routing table, and dev standards
- Each of the 6 workflows has a dedicated spec template
- Any AI agent can follow the templates without requiring the Infrahub Claude plugin
- Existing Copier template functionality is unchanged
